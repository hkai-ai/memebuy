# 梗图图片编辑模板业务使用说明

这份说明给业务、运营、内容和设计同学使用。当前默认目标是先生成 Agent 编辑草稿，再编译成后端 GalleryTemplate 可直接导入的模板 JSON。

## 适用场景

适合使用这套流程的情况：

- 你看到一张梗图，想把它做成前端可编辑模板。
- 你希望用户选择主体、食物、动作、文案或其他变量。
- 你希望用户上传宠物、商品、人物或物件照片作为参考图。
- 你希望大模型先识别图片，并自动生成一些可以替换的候选内容。
- 你需要 mock 用户输入，方便前端预览和产品验收。

不适合直接使用的情况：

- 只想生成最终图片，不关心前端怎么收集用户输入。
- 不需要槽位、候选项或用户可编辑提示词。
- 用户上传图完全看不出主体是什么。

## 推荐工作流

### 先说清楚想要前端编辑模板

最推荐说法：

```text
帮我生成一个前端图片编辑模板。
```

或者更明确：

```text
基于这张图生成 image-edit-template.json。
要求包含用户可编辑提示词、文本槽位、图片槽位、候选项、mock 用户输入和后端拼接提示。
```

业务人员现在重点关注：

- 用户编辑哪些槽位。
- 每个槽位是文字、选择、长提示词、上传图片还是选择图片。
- 是否需要用户上传参考图。
- 是否允许整段重写。
- 候选项是否足够贴近原模板句。
- 后端是否能根据用户最终输入拼接图像编辑指令。

不再默认要求业务选择“高保真还是自由创意”。这两个是历史生成链路里的概念，不是新前端编辑模板的默认入口。

## 交付物

默认输出：

```text
image-edit-template.json
meme-template.json
index.md
```

`image-edit-template.json` 是 Agent 草稿，主要字段：

- `templateText`: 带 `【槽位：默认值】` 的模板句。
- `editablePrompt`: 当前完整提示词，用户可以直接编辑。
- `allowFullRewrite`: 默认 `true`，表示用户可以整段删除、重写或只改槽位。
- `slots[]`: 前端表单槽位。
- `mockUserInput`: 可选示例用户填写结果，默认不生成。
- `backendHint`: 后端拼接建议。

`meme-template.json` 是后端导入文件，固定使用：

- `key/title/description/cover/referenceImage`
- `imageSize/imageN`
- `promptTemplate/inputSchema/preprocessSteps`
- `metadata`

导入脚本按 Schema 校验，上传本地图到 OSS，回填 URL，再按 `key` upsert。前端从后端 API
读取 `inputSchema`，不直接读取本地 JSON。

## 槽位怎么理解

### 文本槽

适合主体、食物、动作、场景、短文案等。

```text
这是一只【主体：狗】在吃【食物：哈密瓜】
```

前端可以让用户把 `狗` 改成 `小猪`，把 `哈密瓜` 改成 `西瓜`。

### 图片槽

适合用户上传或选择图片：

- `identity_reference`: 保留主体身份，例如用户宠物、商品、人物。
- `edit_target`: 直接编辑目标图片。
- `style_reference`: 参考风格、色彩、材质。
- `composition_reference`: 参考构图、镜头、布局。

图片槽必须说明模型需要提取什么，例如主体类别、颜色、轮廓、表情和可稳定复现的身份线索。

### 整段提示词

槽位只是辅助输入。用户可以只改槽位，也可以把整段 `editablePrompt` 改掉，例如：

```text
参考图1，把画面里的狗改成一只穿雨衣的小猪，正在吃西瓜，保持原图构图和可爱风格。
```

这就是 `allowFullRewrite: true` 的含义。

## 候选项

大模型识别图片后，可以为核心文本槽生成 3-8 个候选项。

候选项规则：

- 候选项只是建议，用户可以自定义或清空。
- 候选项必须服务原模板句。
- 不要因为生成候选项而把模板改成另一个梗。
- 如果要保留具体身份，应该用图片槽，而不是只给文字候选。

## Mock 用户输入

mock 用户输入用于前端预览，不是假装真实用户上传。

mock 应包含：

- `slotValues`: 示例槽位值。
- `imageSelections`: 示例图片来源，可能是 `mock_user_upload`、`selected_asset` 或 `none`。
- `renderedTemplateText`: 带新槽位值的模板句。
- `renderedPromptPreview`: 给用户看的提示词预览。

例如：

```text
模板句：这是一只【主体：狗】在吃【食物：哈密瓜】
mock 值：主体=小猪，食物=西瓜
预览：这是一只小猪在吃西瓜
```

## 后端执行字段

后端直接执行 `promptTemplate`、`inputSchema` 和 `preprocessSteps`。展示标记
`【主体：白猫】` 会编译为 `{{ subject | "白猫" }}`。模板固定参考图放
`referenceImage`；构图和锁定约束必须写进 `promptTemplate` 才会影响生成，同时在
`metadata.templateSource` 留档。

用户上传图片默认原图直通生成，`preprocessSteps` 使用 `[]`。只有明确要“读图转文字且不把
原图传给生成模型”时才使用 vision step；运维绑定 `gallery.vision` 前禁止生成该步骤。

## 批量入库

如果目标是后台模板库，而不是前端编辑器，可以继续说：

```text
使用 $meme-template-analyzer 批量分析这个文件夹里的梗图。
请生成可给入库脚本使用的 meme-template.json、batch-manifest.json 和 index.md。
```

批量入库仍关注：

- 批量预审。
- 每张源图独立文件夹。
- 自动聚类。
- `taxonomy`。
- `generationFit`。
- `batch-manifest.json`。
- 批量完成后是否需要查看 `review.html`。
- 下一步选择。

一个模板输出一个 `meme-template.json`。`taxonomy`、模板机制和待审核原因放 `metadata`；
source hash 和追踪状态放 `batch-manifest.json` 或导入后的 `import-report.json`，不进入模板表。

## 批量整理台

如果图片很多，且需要先人工快速归类相似图、勾选标签和确认参考类型，可以单独使用：

```text
生成批量整理台。
```

或：

```text
使用 batch-review-workbench 整理这个文件夹里的图片。
```

整理台是一个静态 `review.html` 小工具，用于：

- 多选相似图片并归到同一组。
- 给每组配置标签和状态。
- 在管理台顶部“标签词库”维护需要人工复用的普通 tags；任务会保存词库快照，AI 标签与平台原始标签另行标记来源。
- 勾选参考类型：`template_reference`、`style_reference`、`composition_reference`、`identity_reference`。
- 标记是否必须使用参考图，例如 `reference_aware_required`。
- 直接写回 `batch-workspace.json`、`batch-manifest.json` 和每个分组的 `group-config.json`。
- 可选复制图片到分组文件夹，原文件不删除。

默认不删除也不移动源文件。该工具需要 Chrome 或 Edge，因为它依赖 File System Access API；不需要 Python 或本地服务。

## 审核页

当需要给业务人员看，可以说：

```text
生成审核页
```

或：

```text
给运营看一下这个模板
```

审核页应展示：

- 核心理解。
- 不应这样理解。
- 必须保留。
- 允许变化。
- 前端槽位。
- `templateText`。
- `editablePrompt`。
- `backendHint`。
- Raw JSON。

## 推荐复制模板

### 前端编辑模板

```text
使用 $meme-template-analyzer 分析这张图，生成前端图片编辑模板。
请输出 image-edit-template.json，并包含：
1. templateText；
2. editablePrompt；
3. allowFullRewrite: true；
4. 文本槽位和图片槽位；
5. 每个文本槽 3-8 个 suggestions；
6. mockUserInput；
7. backendHint。
```

### 用户上传图槽位

```text
请为这个模板增加用户上传图槽位。
说明这个图片槽是 identity_reference、edit_target、style_reference 还是 composition_reference。
每个图片槽必须写 extract、maxCount、private 和 sourceOptions。
```

### Mock 用户输入

```text
请基于这个 image-edit-template.json 生成一组 mock 用户输入。
要求说明 slotValues、imageSelections、renderedTemplateText 和 renderedPromptPreview。
不要把 mock 图片描述成真实用户上传。
```

## 业务验收标准

看 `image-edit-template.json` 时，重点判断：

1. 前端能不能直接渲染表单。
2. 用户是否能理解每个槽位。
3. 用户是否能上传或选择需要的图片。
4. 用户是否可以整段重写提示词。
5. 候选项是否贴近原模板句。
6. mock 用户输入是否能展示真实使用方式。
7. 后端拼接提示是否足够明确。
