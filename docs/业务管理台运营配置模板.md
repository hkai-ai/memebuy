## 1. 最短操作流程

1. 从仓库根目录启动管理台：`pnpm dev`。
2. 打开 `http://127.0.0.1:15173`。
3. 如需复用人工标签，先进入“标签词库”补齐具体 tags；没有需要时可以保持为空。
4. 进入“素材与批次”，选择素材目录并创建批次。
5. 把观看逻辑相同的图片放进同一个分组。
6. 按本文的“分组配置模板”填写每个分组。
7. 先保存配置，再开始生成。
8. 在“任务中心”检查运行状态，在“结果审核”检查 validator、标签和 OSS 状态。

普通模板整理默认选择“只生成模板”，默认不要勾选自动上传 OSS。只有确实需要真实效果图或最终 OSS 交付时，才开启对应选项。

## 2. 标签怎么配置

### 2.1 标签的三种来源

| 来源 | 谁填写 | 示例 | 运营是否直接修改 |
| --- | --- | --- | --- |
| 人工标签 `operator` | 运营在标签词库维护、在分组勾选或结果页补录 | 具体业务 tag | 可以 |
| AI 标签 `ai` | Agent 根据图片自动生成 | 白猫、瞪眼、文字在顶部 | 不需要预先填写 |
| 外部标签 `external` | Pinterest、Instagram、Tumblr 等来源数据 | `funny cat`、`reaction meme` | 保留原文，不当作 AI 标签 |

标签词库不负责维护场景、主题、用途或模板机制。当前包含 8 组风格标签 103 个、4 组情绪标签 16 个；组名只用于整理展示，不会作为 tag 写入模板。

| 分组 | 选择标签所在的整理维度 | 版式·形态 |
| 别名 | 同义词，使用逗号分隔 | collage |

## 3. 素材与分组原则

一个分组应对应一个稳定的模板观看逻辑。仅仅更换人物、动物、食物、颜色或局部文案，一般仍可放在同一分组；如果笑点、主体关系、构图逻辑或用户理解方式发生变化，应拆成不同分组。

分组目录名使用简短、稳定、可识别的英文或拼音，例如：

```text
cat-shocked-reaction
pet-food-fusion
office-text-reaction
```

不要使用“新建分组 1”“待处理”“随便看看”等无法长期识别的名称。

## 4. 分组配置字段说明

### 状态

| 选项 | 适用情况 |
| --- | --- |
| `ready_for_template` | 图片和运营配置已经确认，可以生成 |
| `needs_review` | 文化出处、分类或模板公式仍需人工确认；可以分析，但结果应保留待审核状态 |
| `skipped` | 不处理该分组，不能创建任务 |

### 生成模式

| 选项 | 结果 | 什么时候用 |
| --- | --- | --- |
| 只生成模板 `template` | 生成分析和 JSON，不生成真实效果图 | 默认选择 |
| 分析 + 真实出图 `generation_test` | 在模板分析后生成测试图片 | 用户明确要求看效果时 |

### 模板机制

用一到三句话描述“这个模板为什么成立”，不要只罗列画面元素。

推荐写法：

```text
第一眼把主体看成普通食物，第二眼发现动物身体与食物外形自然融合，反差来自身份误读。
```

不推荐写法：

```text
有一只猫、一个面包、桌子、黄色背景。
```

### 参考图角色

| 选项 | 含义 | 常见选择 |
| --- | --- | --- |
| `template_reference` | 原图定义模板本身和核心梗点 | 大多数参考梗图开启 |
| `composition_reference` | 必须保留位置、构图、遮挡或镜头关系 | 构图是笑点时开启 |
| `style_reference` | 需要继承原图的画风、材质或色彩 | 风格必须相似时开启 |
| `identity_reference` | 需要保留源图人物或主体身份 | 只有身份确实不可替换时开启 |

不要因为图片中出现人物就默认勾选 `identity_reference`。如果终端用户会上传自己的照片替换人物，原图通常是模板/构图参考，而不是身份参考。

### 参考依赖

| 选项 | 含义 |
| --- | --- |
| `low` | 文字描述基本可以独立还原 |
| `medium` | 参考图有帮助，但允许一定构图变化 |
| `high` | 笑点或视觉结构强依赖参考图 |

### 测试建议

| 选项 | 含义 |
| --- | --- |
| `reference_aware_required` | 测试必须携带参考图 |
| `reference_aware_preferred` | 优先携带参考图，必要时可退化 |
| `prompt_mode_allowed` | 纯提示词测试也可以接受 |

推荐组合：

| 素材类型 | 参考图角色 | 参考依赖 | 测试建议 |
| --- | --- | --- | --- |
| 构图固定的经典梗图 | `template_reference` + `composition_reference` | `high` | `reference_aware_required` |
| 风格参考图 | `style_reference`，按需加 `composition_reference` | `medium` | `reference_aware_preferred` |
| 普通反应图 | `template_reference` | `medium` | `reference_aware_preferred` |
| 纯文字创意或不依赖原图 | 不勾选或只保留必要角色 | `low` | `prompt_mode_allowed` |

### Agent 完成后上传 source image 到 OSS

默认关闭。只有同时满足以下条件时才开启：

- 本次任务需要最终 OSS URL 版 JSON。
- 本地 validator 必须先通过。
- OSS 环境变量已经正确配置。
- 用户已经明确授权上传。

如果只是在本地分析和审核，保持关闭。任务完成后也可以在“结果审核”中先检查 OSS 状态，再人工执行重新上传。

### 业务备注

只填写 Agent 无法从图片或结构化字段中得知的业务事实，例如：

```text
该素材来自已授权的七夕专题；文案必须适合情侣聊天回复，不要生成品牌名称。
```

不要在备注中写 Token、AccessKey、密码或其他敏感信息。

## 6. 完整填写示例

```text
分组目录名：cat-shocked-reaction
状态：ready_for_template
生成模式：template

人工标签：
- （按当前业务需要填写具体 tag；本例无）

模板机制：
猫保持呆住或震惊的表情，画面上方短文案与主体反应形成对应。笑点来自文案语境和过度反应之间的反差。

参考图角色：
- template_reference：是
- composition_reference：是
- style_reference：否
- identity_reference：否

参考依赖：medium
测试建议：reference_aware_preferred
Agent 完成后上传 source image 到 OSS：否

业务备注：
用于聊天回复；文案保持短句，不出现品牌名和网络攻击内容。
```

AI 可以在分析后补充“白猫”“瞪眼”“正面近景”“文字在顶部”等细标签，运营不需要提前把这些全部录入词库。

## 7. 结果审核清单

任务完成后逐项确认：

- semantic validator 和 Gallery validator 是否为 `PASS`。
- `needsReview` 是否存在，存在时是否完成了人工确认。
- 人工选择的普通标签是否被保留。
- AI 标签是否准确，是否出现把普通画面元素过度标签化的问题。
- 人工二次标签是否只修改 `manual/operator` 标签。
- `cover` 和 `referenceImage` 是否指向正确的 source image。
- 需要 OSS 交付时，对象是否真实存在，而不只是 JSON 中有 URL。
- 不需要 OSS 交付时，是否避免了无意义上传。

## 8. 给 Agent 的操作请求模板

把下面内容复制给 Agent，并替换尖括号中的值：

```text
请按照 AGENTS.md 和 docs/业务管理台运营配置模板.md 启动并操作本地 Meme 业务管理台。

素材目录：<绝对路径>
批次名称：<批次名称>
分组策略：<按什么观看逻辑分组>

本批次人工标签：<具体 tags；没有则写“无”>
生成模式：只生成模板 / 分析并真实出图
是否允许上传 OSS：是 / 否

特殊业务要求：<授权、受众、文案、版权或审核要求>

启动前检查端口，后台运行并完成 API 探活。不要部署到公网，不要提交 artifacts 或 .meme-admin。
完成后报告批次、分组、任务状态、validator、标签和 OSS 检查结果。
```

Agent 不应为了填满分类维度而创建标签；保留可靠的 AI 细标签即可，人工词库由运营按实际业务需要维护。
